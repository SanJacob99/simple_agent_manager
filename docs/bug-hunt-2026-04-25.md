# Bug Hunt Report — 2026-04-25

## Scope and method

This pass covered the full repository with two high-signal checks:

1. **Full unit/integration test run**: `npm run test:run`
2. **Full production build + typecheck**: `npm run build`

The build succeeded, but the test run found **8 regressions** across runtime logging behavior and store/session behavior.

---

## Findings

### 1) `clearSessionMessages()` no longer resets session identity (regression)

- **Symptom:** Clearing an active transcript does not call backend `resetSession()` and therefore does not guarantee a fresh session ID/lineage.
- **Evidence:** `src/store/session-store.test.ts` expects `resetSession(sessionKey)`; call count is currently `0`.
- **Code path:** `clearSessionMessages()` now calls `storageEngine.clearSessionMessages(sessionKey)` and hydrates via `getSession()`.
- **Risk:** Session lineage and context reset semantics can drift from UI expectation, especially for users who treat "clear chat" as a hard conversation reset.

### 2) Prompt debug logging can block progress when clone body read is slow

- **Symptom:** Debug payload logging path may await response clone body read in a way that delays prompt resolution.
- **Evidence:** `server/runtime/agent-runtime.test.ts` (`does not block prompt progress while debug body logging reads the response clone`) fails because `resolved` stays `false`.
- **Risk:** User-visible latency spikes when debug logging is enabled and response body inspection is slow.

### 3) Payload summary log contract drift (reasoning fields injected)

- **Symptom:** Request payload summary now always includes `reasoning`, `reasoning_effort`, and `keys=[...]`, while tests and downstream parsers previously expected a shorter stable format.
- **Evidence:** 5 failing assertions in `server/runtime/agent-runtime.test.ts` around exact expected summary strings.
- **Risk:** Log-based parsing/monitoring and deterministic test baselines break.

### 4) Tools node default profile mismatch between graph-store expectation and defaults

- **Symptom:** New `tools` nodes initialize with `profile: "custom"` while store integration test expects `"full"` defaults behavior.
- **Evidence:** `src/store/graph-store.test.ts` failure `Expected: "full" Received: "custom"`.
- **Risk:** New graphs may start in unexpectedly restrictive tool mode, depending on intended product default.

### 5) Test suite and implementation are currently out of sync in at least three subsystem contracts

- **Subsystems impacted:**
  - runtime payload logging contract (`server/runtime/*`)
  - session clearing/reset contract (`src/store/session-store.ts` + storage client/server routes)
  - graph default contract for tools profile (`src/utils/default-nodes.ts` + `src/store/graph-store.ts`)
- **Risk:** Hidden behavior changes can ship because build passes but expected product semantics are no longer guarded by passing tests.

## Recommended triage order

1. **Decide intended clear-chat semantics** (`clear` vs hard `reset`) and align store + tests + backend route naming.
2. **Ensure debug logging is non-blocking** by decoupling clone body read from request critical path.
3. **Stabilize payload summary contract** (or version it) to avoid brittle downstream parsing.
4. **Finalize intended default tools profile** (`custom` vs `full`) and align tests + UX copy.
5. Re-run full suite and require green tests before merge.
